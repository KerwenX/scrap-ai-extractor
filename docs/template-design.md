# 模板设计说明

## 1. 设计目标

本项目中的“模板”不是手写站点 parser，也不是某个页面的一次性脚本。

模板的本质是：
- 某一类页面结构的可执行抽取方法
- 可落盘的 JSON 资产
- 可版本化、可迁移、可复用的 DSL 规则集合

LLM 负责第一次理解页面；模板负责把这次理解沉淀成后续可直接复用的确定性能力。

## 2. 为什么不用手写站点 parser

如果继续堆手写 parser，很快会出现这些问题：

- 同站点页面一多，代码里会充满 `if/else`
- 规则升级必须改代码、发版、回滚
- 规则很难迁移到别的机器或别的语言项目
- LLM 首次成功抽取后，无法自然沉淀成可复用资产

所以当前方向是：

- 核心代码尽量不写页面定制逻辑
- 抽取规则尽量沉淀到模板 `JSON + DSL`
- 通用能力沉淀到运行时算子和匹配策略

## 3. 模板分层

### 3.1 正式模板

正式模板对应 `TemplateManifest`，用于直接运行。

关键字段：
- `template_id`
- `template_key`
- `site_id`
- `scenario`
- `page_type`
- `version`
- `lifecycle_status`
- `fingerprint`
- `required_fields`
- `extraction_plan`

其中：
- `template_key` 表示模板家族
- `template_id` 表示模板家族下的具体版本

### 3.2 候选模板

候选模板对应 `TemplateCandidate`，用于记录一次 LLM 成功抽取后的中间结果。

关键字段：
- `fingerprint`
- `extracted_fields`
- `sample_data`
- `analysis`
- `proposed_plan`

候选模板不是最终模板，但它提供了固化所需的证据。

## 4. 模板命名思路

模板命名不能只靠页面标题，也不能直接绑定单个页面。

当前推荐思路：

- `template_key` 以站点、场景、页面类型、URL 结构摘要为主
- `template_id = template_key + version`
- DOM 指纹用于辅助匹配与版本演化，不直接等同于模板家族标识

这样做的目的，是让“同站点、同结构、不同内容”的页面优先复用同一模板家族，而不是为每个页面生成一套新模板。

## 5. 模板命中策略

模板命中不能只靠 DOM 指纹。

当前 DSL 模板命中采用综合评分：

1. 先按 `site_id` 过滤
2. 再看 `scenario / page_type`
3. 对模板逐个试跑
4. 统计运行时字段命中情况
5. 结合页面指纹相似度计算综合分数

关键评分项：

- `fingerprint_score`
- `selector_hit_rate`
- `required_hit_rate`
- `classification_affinity`
- `match_score`

这比“只看 DOM 指纹”更适合一个站点下存在多个相似模板的场景。

## 6. URL 在模板命中的作用

当前模板体系是“站点优先”的：

- 模板首先按 `site_id` 过滤
- `site_id` 主要来自 URL 的 host
- 所以在只允许模板解析的模式下，`url` 是必填项

这也是为什么：

- 单条自动解析可以不传 `url`
  - 因为它允许后续回退到 LLM
- 单条仅模板解析和批量解析必须传 `url`
  - 因为这两种模式不允许回退到 LLM，必须在前面就把模板命中边界说清楚

当前系统行为：

- `run_mode=auto`
  - 可以不传 `url`
  - 未命中模板时可回退到 LLM
- `run_mode=template_only`
  - 必须传 `url`
  - 未命中模板时直接失败

## 7. DSL 的职责边界

### 7.1 字段定位

DSL 当前支持的选择器类型包括：

- `css`
- `id`
- `meta`
- `text_pattern`
- `section_tab`
- `label_value`
- `code`

其中 `label_value` 适用于这类常见结构：

```html
<tr><td>作者</td><td>张三</td></tr>
<tr><td>医院</td><td>北京协和医院</td></tr>
```

这类定位方式具有明显的跨页面复用价值，不属于站点定制逻辑。

### 7.2 轻量后处理

后处理算子包括：

- `strip`
- `strip_cn_punctuation`
- `split_cn_list`
- `unique`
- `first_non_empty_line`
- `regex_extract`
- `regex_replace`
- `join`
- `filter_empty`
- `normalize_whitespace`
- `to_int`
- `to_float`

原则是：

- 页面级规则放模板里
- 通用处理模式放运行时算子里
- 业务代码尽量不为具体站点改逻辑

## 8. 内容后处理能否完全记录在模板里

结论是：大部分常见后处理可以模板化，但不是所有业务语义都适合直接写成页面定制代码。

更合理的做法是分两层：

### 第一层：模板内声明式后处理

适合处理：

- 去空白
- 拆分列表
- 正则提取
- 正则替换
- 去重
- 类型转换
- 拼接和过滤

这类处理适合由 DSL 完整记录，并且跨语言迁移成本低。

### 第二层：运行时通用算子

如果后续出现更复杂的处理，也应该优先沉淀成“通用算子”，而不是改某个站点专用逻辑。

也就是说：

- 不建议在核心代码里给某个站点写特殊后处理
- 建议把复杂但通用的处理沉淀成少量固定范式
- 模板只负责声明“用哪个算子”和“传什么参数”

这样可以尽量做到：

- 代码层面不需要为新站点改逻辑
- 模板层面可以持续扩展能力

## 9. 模板闭环

当前模板闭环收敛为三种动作：

- `create`
  - 没有相同模板家族，创建新模板
- `upgrade`
  - 已存在相近正式模板，但当前候选规则更完整，生成新版本并停用旧版本
- `reuse`
  - 既有模板已覆盖当前能力，不再重复晋升

这样可以避免以前那种反直觉情况：

- 运行时走了 LLM
- 候选模板却因为“已存在模板”而无法晋升

现在更合理的闭环应该是：

- 候选更强，就升级正式模板
- 候选不更强，就复用现有模板

## 10. 为什么模板适合迁移

模板当前以文件形式落盘在：

- `config/templates/`
- `data/template_store/`
- `data/template_candidates/`

这意味着迁移时只需要带上：

- 代码
- 配置
- 模板文件

而不依赖某个数据库里的隐式状态。

这也是后续给 Java 运行时做轻量模板消费者的基础。

## 11. 一句话总结

模板不是某个页面的临时解析脚本，而是某类页面结构化抽取方法的版本化资产。
