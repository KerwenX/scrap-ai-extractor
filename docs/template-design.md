# 模板设计说明

## 1. 模板的定义

本项目里的“模板”不是手写站点 parser，也不是某个页面的一次性脚本。

模板的本质是：

- 某一类页面结构的可执行抽取契约
- 可落盘的 JSON 资产
- 可版本化、可迁移、可回放的 DSL 规则集合

模板的目标不是“让当前这个页面跑通”，而是“把一次成功抽取沉淀成后续可复用的确定性能力”。

## 2. 为什么模板不能等于手写 parser

如果继续堆手写 parser，会很快遇到这些问题：

- 同站点页面一多，代码里会出现大量 `if/else`
- 规则升级必须改代码、发版、回滚
- 规则无法轻量迁移到别的机器或别的语言项目
- LLM 首次成功抽取后，很难自然沉淀成后续复用能力

所以当前方向是：

- 核心代码尽量不写页面定制逻辑
- 抽取规则尽量沉淀到模板 JSON + DSL
- 通用能力沉淀到运行时算子和匹配策略

## 3. 模板的两种形态

### 3.1 正式模板

正式模板对应 `TemplateManifest`，用于直接运行。

关键字段：

- `template_id`
- `template_key`
- `site_id`
- `scenario`
- `page_type`
- `version`
- `lifecycle_status`
- `fingerprint`
- `required_fields`
- `extraction_plan`

其中：

- `template_key` 表示模板家族
- `template_id` 表示模板家族下的某个具体版本

### 3.2 候选模板

候选模板对应 `TemplateCandidate`，用于记录一次 LLM 成功抽取后的中间结果。

关键字段：

- `fingerprint`
- `extracted_fields`
- `sample_data`
- `analysis`
- `proposed_plan`

候选模板不是最终模板，但它提供了固化所需的证据。

## 4. 为什么要做“模板家族”而不是“样本页面模板”

如果模板 key 直接绑定某个页面的 DOM 指纹，那么结果就是：

- 同结构不同内容页面无法复用
- 每个页面都会生成自己的模板
- 模板数量快速膨胀

这违背模板系统的目标。

所以当前已经收敛到：

- 自动固化时，优先生成模板家族 key
- `template_key` 以站点、场景、页面类型、URL 模式摘要为主
- DOM 指纹只作为识别与版本演化的证据，不再直接决定“是不是一个新家族”

## 5. 模板命中策略

模板命中不能只靠指纹。

对于同站点页面，更合理的判断顺序是：

1. 先按 `site_id` 过滤模板集合
2. 再看 `scenario/page_type`，作为分类亲和度先验
3. 对候选 DSL 模板逐个试跑
4. 计算运行时命中率与必填覆盖率
5. 再结合指纹相似度给出综合分
6. 选择得分最高的模板

当前实现中的关键分数：

- `fingerprint_score`
- `selector_hit_rate`
- `required_hit_rate`
- `classification_affinity`
- `match_score`

强匹配的经验条件：

- `selector_hit_rate >= 0.85`
- `required_hit_rate >= 0.85`

如果多个模板都超过这个区间，则直接选择 `match_score` 最高的模板。

## 6. 为什么运行时命中率比纯指纹更重要

指纹解决的是“像不像”。

运行时命中率解决的是“能不能真的抽出来”。

实际工程里后者更关键。因为：

- 同站点不同栏目可能 DOM 很像，但字段位置不同
- DOM 轻微改版会让指纹下降，但规则仍然有效
- 真正要执行的是模板，不是指纹本身

因此对于正式 DSL 模板，当前策略是：

- 优先看运行时字段命中结果
- 指纹作为辅助证据
- 指纹低但字段覆盖稳定时，不直接回退 LLM

## 7. 通用定位器设计

当前 DSL 选择器包括：

- `css`
- `id`
- `meta`
- `text_pattern`
- `section_tab`
- `label_value`
- `code`

其中 `label_value` 是这轮新增的重要能力。

适用结构：

```html
<tr><td>作者</td><td>张三</td></tr>
<tr><td>医院</td><td>北京协和医院</td></tr>
```

这个定位器是通用能力，不属于站点定制逻辑。

它的意义是：

- 不要求值节点有唯一 `id/class`
- 更适合论文页、医生页、商品页等详情页
- 更容易在首次 LLM 成功后自动固化成可复用模板

## 8. 后处理为什么放到 DSL 而不是业务代码

字段抽取之后，通常还需要做轻量处理。

当前通用后处理算子包括：

- `strip`
- `strip_cn_punctuation`
- `split_cn_list`
- `unique`
- `first_non_empty_line`
- `regex_extract`
- `regex_replace`
- `join`
- `filter_empty`
- `normalize_whitespace`
- `to_int`
- `to_float`

设计原则是：

- 页面定制逻辑尽量不要写进业务代码
- 通用处理模式尽量沉淀成 DSL 算子
- 模板只声明“用哪个算子”，运行时统一执行

这也是模板可迁移、可跨语言复用的前提。

## 9. 自动固化策略

当前自动固化不是“LLM 一成功就直接发版模板”，而是更保守：

- 候选模板必须有 `proposed_plan`
- 计划里必须真的有字段规则
- `required_fields` 必须和 DSL 可覆盖字段尽量对齐
- 对于已存在同指纹正式模板，优先复用而不是重复创建

另外这轮已经修复了一个关键问题：

- 以前 `required_fields` 可能远多于 DSL 真正覆盖的字段，导致模板一运行就自我校验失败
- 现在会优先对齐到 DSL 实际可覆盖字段，避免“刚固化就回退”

## 10. 模板为什么适合迁移

模板当前以文件形式落盘在：

- `config/templates/`
- `data/template_store/`
- `data/template_candidates/`

这意味着迁移时只需要带上：

- 代码
- 配置
- 模板文件

而不需要依赖某个数据库里的隐式状态。

这也是后续给 Java 运行时做轻量模板消费器的基础。

## 11. 当前设计边界

当前项目明确不做两类事：

1. 不做爬虫抓取
2. 不在核心代码中持续增加页面定制化逻辑

真正允许扩展的只有两类：

- 新的通用 DSL 选择器
- 新的通用后处理算子

这能保证系统继续沿着“通用抽取基础设施”演进，而不是退化成“站点脚本仓库”。

## 12. 一句话总结

模板不是某个页面的临时解析脚本，而是某类页面结构化抽取方法的版本化资产。

LLM 负责第一次理解页面，模板负责把这种理解沉淀成后续可直接复用的确定性能力。
