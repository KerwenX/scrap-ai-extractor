# 需求文档

## 1. 背景

目标是构建一个面向网页数据抽取的混合解析系统。

系统只负责“解析”，不负责“抓取”。调用方需要自己提供：

- `url`
- `raw_html`
- `user_prompt`

系统根据输入完成结构化抽取，并在成本、性能、稳定性和可迁移性之间取得平衡。

## 2. 产品目标

- 支持调用方通过自然语言描述抽取目标
- 对首次出现的页面模板，使用 LLM 进行语义理解和结构化抽取
- 对已经识别并固化过的模板，优先使用确定性解析规则
- 当确定性解析失败或模板漂移时，自动回退到 LLM
- 为后续模板固化保留运行时证据、候选规则和调试链路

## 3. 输入与输出

### 输入

- `url`
- `raw_html`
- `user_prompt`

### 输出

- `data`
- `template_id`
- `extractor_type`
- `validation_report`
- `confidence`
- `drift_detected`
- `debug_trace`

## 4. MVP 范围

### 当前支持

- 单页 HTML 解析
- 中文网页解析
- 站点识别和页面场景识别
- 已知模板的确定性解析
- 未知模板或失败场景下的 LLM 回退
- 模板候选的持久化
- 请求级日志
- 本地 CLI 和本地 API

### 当前不支持

- 列表翻页抓取
- 登录态采集
- 多页面联动抽取
- 自动发布模板规则
- 后台任务调度

## 5. 功能需求

### 5.1 页面预处理

- 清理脚本、样式、噪声标签
- 提取页面标题、描述和主体文本
- 保留足够的 DOM 结构用于模板识别和规则抽取

### 5.2 意图理解

- 将 `user_prompt` 映射为内部抽取意图
- 识别常见页面类型和字段需求
- 将用户业务目标与系统内部提示词约束解耦

### 5.3 模板识别

- 识别站点
- 识别页面场景
- 结合页面指纹匹配已固化模板
- 返回稳定的 `template_id`

### 5.4 确定性解析

- 根据模板和规则执行字段抽取
- 支持声明式 DSL 规则
- 在必要时保留代码型解析器兜底

### 5.5 LLM 回退

- 未命中模板时走 LLM 抽取
- 确定性解析校验失败时走 LLM 回退
- 抽取结果必须可校验、可追踪

### 5.6 结果校验

- 必填字段校验
- 字段覆盖率统计
- 输出漂移信号
- 区分成功、失败和回退路径

### 5.7 模板固化

- 允许将成功的回退结果保存为候选模板
- 支持从候选模板逐步固化为正式模板
- 固化结果应可迁移到别的机器直接使用

### 5.8 接口能力

- 提供本地 CLI
- 提供本地 HTTP API
- 提供简单友好的本地 Web 界面
- 返回统一的响应结构

## 6. 非功能需求

- 已知模板优先低成本执行
- 模板规则与业务代码分离
- 代码结构清晰，便于后续扩展
- 结果可追踪，可回放，可调试
- 模板和候选结果可以文件化迁移

## 7. 模板固化运行时契约

当 LLM 回退成功且页面具有复用价值时，系统需要保留两阶段运行时产物：

### 第一阶段：模板分析

输出内容包括：

- 字段形态
- 可能的锚点
- 可复用的结构线索
- 不适合确定性抽取的字段

### 第二阶段：DSL 计划

输出内容包括：

- 可复用的声明式抽取规则
- 规则适用字段
- 需要保留为 LLM 回退的字段

这不是单纯的文档约定，而是运行时数据契约。

## 8. 配置需求

- API Key 等敏感信息不得直接提交到 GitHub
- 项目仓库中保留 `config/app_config.template.json`
- 本地真实配置使用 `config/app_config.json`
- 允许环境变量覆盖本地配置
